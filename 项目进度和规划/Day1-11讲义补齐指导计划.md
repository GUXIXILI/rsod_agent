　　仓库管理员负责主干分支保护与 PR 审核合并。PR 审核遵循 P0、P1、P2 分级策略，P0 问题阻塞合并直到修复完成，大多数 P1 问题同样阻塞合并，P2 问题仅作提醒不阻塞。

　　合并前需确认提交者是否在 PR 描述中标注提醒目标，即负责某某角色的人，以便后续联调跟进。
　　对完全无价值的分支需及时关闭删除，参考 7 月 15 日已关闭 5 个重复分支的操作。

## 十一、Day11 补齐计划

　　Day11 讲义标题为 Agent 优化加 RAG 加记忆增强，分 Prompt 优化、SSE 流式完善、对话记忆、RAG 知识库、多工具 Agent、前端可视化、LangGraph 多 Agent 七个维度。其中前六个为必做内容，LangGraph 为进阶选做。

　　Prompt 模板优化与 Tool 定义增强维度。需新建 app/agent/prompts.py 集中管理所有提示词，包括检测 Agent 系统提示词（声明 4 种能力、工具调用优先级规则、回复格式要求）、RAG 问答提示词（基于检索内容回答、不编造信息）、Supervisor 调度提示词（可选）。需新建 app/agent/tools/ 目录，将检测工具从 detection_agent 拆分为独立的 detection_tool.py，包含 4 个检测工具函数，每个使用 @tool 装饰器并配有详细 docstring 说明用途、参数、返回值。该维度责任角色为 AI 算法工作人员。请查看 Day11 讲义第三节内容。

　　SSE 流式响应完善维度。需将 SSE 事件协议从 2 种扩展为 6 种，包括 thinking（思考中）、tool_start（工具开始调用）、tool_end（工具调用完成）、text_chunk（文本片段）、done（完成）、error（错误）。需在 chat_service 的流式生成逻辑中按实际执行流程发出对应事件。思考阶段发 thinking，调用工具前发 tool_start、工具返回后发 tool_end，生成回复文本时逐段发 text_chunk，全部完成发 done，出错发 error。需保持向后兼容，旧的 text_chunk 事件格式不变。该维度责任角色为后端工作人员配合 AI 算法。请查看 Day11 讲义第四节内容。

　　对话记忆管理维度。需新建 app/agent/memory.py，实现 ConversationMemory 类，基于 Redis List 存储对话历史。key 格式为 chat:session:{user_id}:{session_id}，value 为 JSON 序列化的消息列表，TTL 为 3600 秒即 1 小时。需支持 save_message 保存消息、load_history 加载历史（最多 20 条）、get_sessions 获取所有会话、clear_session 清空会话。需在 chat_stream 流程中集成，Agent 调用前加载历史消息作为上下文，用户消息与 AI 回复都追加到记忆中。需提供降级策略，Redis 不可用时回退到内存缓存。该维度责任角色为后端工作人员。请查看 Day11 讲义第五节内容。

　　RAG 知识库模块维度。需新建 backend/knowledge_base/ 目录，预置 3 篇知识文档，包括 yolo_basics.md（YOLO 基础知识、IoU、NMS、mAP 等）、remote_sensing.md（遥感目标检测知识）、evaluation_metrics.md（模型评估指标详解）。需新建 app/rag/ 目录，实现 document_loader.py 文档加载与分块（chunk_size=500、chunk_overlap=50）、embedding.py 文本向量化服务（支持通义千问与 OpenAI Embedding）。需新建 app/vectorstore/pgvector_client.py，实现 knowledge_embeddings 表管理、向量插入、余弦相似度检索。需新建 app/api/knowledge.py 知识库管理 API（上传文档、重建索引、检索测试）。需在应用启动时自动加载知识库文档并构建向量索引。该维度责任角色为后端工作人员配合 AI 算法。请查看 Day11 讲义第六节内容。

　　多工具 Agent 升级维度。需将单 Agent 绑定的工具从 4 个检测工具扩展到 8 个以上，新增 search_knowledge（RAG 知识检索）、query_detection_stats（检测统计查询）、query_detection_history（历史记录查询）、query_user_list（用户列表查询）。需重构 Agent 系统提示词，明确列出所有工具的用途与调用优先级规则。需确保 Agent 能根据用户输入的自然语言正确选择工具，例如询问专业知识时调用 RAG 工具、询问统计数据时调用统计工具。需在 LLM_STUB_MODE 为 true 时，所有工具的调用结果使用模拟数据返回，不发起真实远程 API 调用。该维度责任角色为 AI 算法工作人员配合后端。请查看 Day11 讲义第七节内容。

　　前端 ChatPage 流式渲染与工具可视化维度。需重写 ChatPage.vue 以支持完整的六种 SSE 事件渲染。thinking 事件显示思考中动画与提示文字。tool_start 事件显示当前调用的工具名称与加载状态。tool_end 事件展示工具返回结果的摘要（可折叠展开详情）。text_chunk 事件逐字追加到回复气泡中。done 事件结束加载状态。error 事件显示错误信息与重试按钮。需新增工具调用过程的时间线展示，用户可以看到 Agent 思考、调用工具、获得结果、生成回复的完整流程。需配合 stream.js 工具的事件解析增强，支持新事件类型。需新增会话列表侧边栏，支持创建新会话、切换历史会话。该维度责任角色为前端工作人员。请查看 Day11 讲义第九节内容。

　　LangGraph 多 Agent 编排维度（进阶选做）。能力充足的成员可挑战此项。需新建 app/agent/state.py 定义 AgentState 状态结构。需新建 app/agent/supervisor.py 实现 Supervisor Agent 路由决策。需新建 app/agent/agents/ 目录，实现 detection_agent.py、analysis_agent.py、qa_agent.py 三个子 Agent。需新建 app/agent/graph.py 使用 LangGraph 构建状态图，实现 Supervisor 路由到子 Agent、子 Agent 执行后返回结果的循环流程。该维度责任角色为 AI 算法工作人员。请查看 Day11 讲义第八节内容。

　　依赖关系方面，SSE 流式完善依赖 Prompt 与 Tool 拆分完成，对话记忆依赖后端 Redis 已就绪，RAG 知识库依赖 pgvector 扩展已启用（docker-compose 中已配置），多工具 Agent 依赖前四个维度全部完成，前端可视化依赖后端 SSE 事件协议稳定，LangGraph 为可选不阻塞主流程。建议按 Prompt 优化 → SSE 协议 → 对话记忆 → RAG 知识库 → 多工具 Agent → 前端可视化的顺序推进，前后端可并行。

## 十三、注意事项与风险提示

　　第一，LLM_STUB_MODE 等于 true 约束下大模型调用必须使用占位实现，不主动发起真实远程调用，避免触发外部 API 计费或合规风险。RAG 的 Embedding 同样受此约束，占位模式下可使用随机向量或简单词向量替代。

　　第四，测试时已结束的测试及时关闭，控制内存占用避免电脑死机。
　　第五，使用 trae 内置浏览器测试，不使用电脑浏览器软件。
　　第六，不安装未经许可的软件。
　　第七，提交前先拉取最新主干，在自己分支上解决冲突。Day11 新增代码量较大，建议多次小步提交而非一次大提交。

　　第九，不要提交连自己都没跑过的代码。
　　第十，不要把大模型真实 API 调用塞进占位符位置。
　　第十一，Day11 各模块之间依赖较强，开发前请先通读 Day11 讲义全文了解整体架构，再动手写代码，避免返工。
