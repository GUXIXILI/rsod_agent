# rsod_agent 项目 7_15 工程计划（晚间修正版）

## 一、前言

　　本文档用于指导 rsod_agent 项目在 7 月 15 日之后较长一段周期内的工程推进工作，目的有三个：第一，明确当前工程做到了什么程度、与最终目标的差距在哪里；第二，规定未来要完成的内容以及详细的代码框架与接口约束，防止再次出现组员提交内容与主干完全不适配、无法合并的情况；第三，给各角色（项目经理、Git 管理员、AI 工程师、后端、前端、测试）划定任务边界与最低完成标准。当前工程主分支 main 已通过合并请求 PR #16 合并了 7 月 14 日晚 20 点之前所有分支的内容，由仓库管理员修复了各版本之间的 bug 与适配问题，已具备用户认证、单图与批量与视频检测、历史记录、数据看板、训练服务、火灾预警、健康检查、日志监控、数据集转换、Docker 容器化部署（后端 Dockerfile、前端 Dockerfile、nginx.conf、docker-compose.yml）等基础能力。7 月 15 日白天管理员又通过 PR #18 上传了 7_15 早间工程进度与工程计划文档，紧接着通过 commit 11170eb 基于只增不减策略吸收了所有成员分支的增量成果，共修改 21 个文件，新增前端摄像头实时检测页面、模型评估导出服务、视频检测 API、ONNX 导出依赖等内容，main HEAD 推进至 11170eb。4 个完全合并的祖先分支（docs/project-progress-and-planning、hb956-pop-patch-1、integration/all-branches-demo、yubai）已标记为旧版本，打上 git 标签 old-version/分支名，并通过 Issue #19 归档关闭，后续合并工作不再考虑这些分支。

　　本次 11170eb 吸收提交遵循的核心原则有三条：第一，纯增量内容直接吸收，例如 PR #17 的模型评估导出服务、新前端分支的摄像头检测页面，都属于纯增量内容，直接吸收到主干；第二，同名页面冲突时新开独立页面实现双版本共存，例如 DetectionPage.vue 与 CameraDetectionPage.vue 同时存在，路由分别为 /detection 与 /camera-detection，让两个版本在实际运行中比较择优；第三，纯删除与配置替换内容不吸收，例如 PR #8 中删除已合并文件的改动、PR #11 至 #15 中与主干重复的 Dockerfile 与 nginx.conf 等配置文件，均不吸收。这次吸收后，主干功能进一步完善，但同时也带来了双版本共存页面需要在实际使用中比较择优的新任务。

　　当前仓库共有 9 个独立分支仍可被后续合并工作考虑，分别是 feature/frontend（PR #8，前端对话页面重写与训练页字段修复，有 3 个 P0 问题待修复，已审核 Request Changes）、feature/model-evaluation-export-yubai（PR #17，模型评估与导出工作流，状态 CONFLICTING，核心内容已被 11170eb 吸收，作者可选择关闭或 rebase 后保留作为补充）、frontend（基于成悦的前端代码新增的第九天前端内容，无 PR，部分内容已被 11170eb 吸收，Issue #20 给出审核报告）、hb956-pop-patch-2 到 hb956-pop-patch-7（PR #11 到 PR #15，容器化部署相关文件，部分有冲突，全部已审核 Request Changes）。所有 7 个 open PR 都已经过管理员审核并发出 Request Changes，作者需要按审核意见修复后重新提交。当前工程存在的主要问题是：很多组员提交的内容与主干接口完全不适配，没有修复 bug，也没有听取 AI 的改进建议，导致提交内容完全无法合并到主干，只有部分内容可取。本文档就是为解决这一现象而制定，后续所有提交必须严格遵守本文档规定的接口契约与提交规范。

　　需要特别说明的是大模型占位符约束。当前工程设置环境变量 LLM_STUB_MODE=true，所有大模型调用必须使用占位实现，不主动发起真实远程调用，主要原因是当前经费不足，避免产生额外接口调用费用。如果有人愿意自费配置真实大模型接口并承担相应费用，可以支持接入，但接入真实接口的代码仍需通过合并请求审核，确保不泄露密钥、不影响现有功能、不引入需要付费但未声明的依赖。前端方向的同学需要单独说明：前端页面布局、样式、交互细节由前端自由发挥，本计划只规定接口必须对齐的边界，不规定具体的视觉风格。前端应在完成功能基础上优化已有界面使其更美观，包括配色方案、字体规范、间距留白、过渡动画、加载状态、表单表格视觉、暗色与亮色模式一致性、响应式适配等，如果前端做的东西和后端不匹配，应主动让后端去改，不要在那里什么都不干。

## 三、当前工程进度总览

### 三之一 已完成内容

　　经过前期开发，以下模块已经达到可用或可运行状态。环境搭建方面，基础设施三件套加 pgvector 可以正常启动，后端服务与前端服务均已通过 docker-compose 一键拉起，前端 nginx 反向代理已正确配置 API 与 SSE 流式传输。后端项目基于 FastAPI 加 SQLAlchemy 加 Alembic 完成，用户认证、权限控制、审计日志、请求日志、对象存储封装、定时任务调度等基础设施已经可用。前端项目基于 Vue 3 加 Vite 加 Element Plus 加 Pinia 完成，登录页、注册页、检测工作台页、摄像头检测页、历史记录页、训练页、数据看板页、智能对话页（占位）已经存在并可正常访问，路由已默认重定向到对话页，11170eb 新增 /camera-detection 路由与摄像头检测菜单项。

　　数据与模型方面，数据集格式转换工具已经齐全，支持 COCO、LabelMe、VOC 三种主流标注格式向 YOLO 格式转换。模型训练服务已经实现完整的训练流程，支持启动训练任务、记录训练指标、保存模型权重、停止训练、下载结果、测试图预测，11170eb 新增 3 个路由（评估、导出、下载）与 model_evaluation_service、model_artifact_service 两个服务，支持模型评估、ONNX 导出与下载。检测服务已经支持单张图片检测、批量图片检测（最多 20 张）和视频检测（含跳帧与多帧确认机制），检测结果可以保存到数据库并上传到 MinIO 对象存储，火情等级判定（safe、notice、warning、danger）已可用，火灾预警记录自动生成并存入 fire_alerts 表，11170eb 在前端 detection.js 新增 detectVideo 与 getVideoStatus API、在 ChatPage.vue 新增视频按钮可点击与 handleVideoDetect 与 pollVideoProgress 函数。历史记录查询、数据统计看板（总览、火情等级分布、检测趋势）等模块也已经可用。

　　前端摄像头检测页面方面，11170eb 新增 CameraDetectionPage.vue（604 行），路由 /camera-detection，与 DetectionPage.vue 双版本共存，让两个版本在实际运行中比较择优。这是只增不减策略的体现，新页面与旧页面同时存在，接入相同的后端接口（待后端摄像头接入层实现），让两个页面都可以正常运行与被测试，最终选择效果更好的一个保留。前端检测结果卡片组件 DetectionResultCard.vue 在 11170eb 中新增视频检测结果展示模块，含视频播放器、信息标签、关键帧网格。后端评估工具 evaluate_fire_smoke_model.py（96 行）已新增，可作为独立评估脚本使用，火灾烟雾模型评估报告.md（92 行）已新增，记录模型评估结果。

### 三之二 部分完成内容

　　以下模块已有基础实现，但尚未达到完整可用状态。模型评估与导出模块方面，11170eb 吸收 PR #17 核心内容后已基本完成，包括 model_evaluation_service.py（345 行）、model_artifact_service.py（118 行）、evaluate_fire_smoke_model.py（96 行）、3 个新路由（评估、导出、下载）、5 个 Pydantic schema 类、onnx 与 onnxscript 依赖、6 个测试文件，但前端展示页面与导出下载入口仍需完善，ONNX 导出能力的实际验证仍需进行。智能对话模块方面，后端已经实现了基于 SSE 流式传输的消息接口（事件类型 start、token、done、error），会话增删改查已实现，11170eb 在 ChatPage.vue 新增视频按钮可点击与 handleVideoDetect 与 pollVideoProgress 函数，但还没有智能体路由分发能力，目前为单 LLM 占位回复，agent_used 字段统一标记为 supervisor，工具调用字段 tool_calls 与 tool_result 未正确填充。检测接口方面，单图、批量、视频、视频帧、推理、预警接口已实现，但压缩包检测、摄像头实时检测未实现，前端 detection.js 中 detectZip 函数已封装但后端无对应实现，11170eb 新增的 detectVideo 与 getVideoStatus API 已可用。

## 四、差距分析

### 四之一 当前网站与最终目标网站的区别

　　当前网站只能检测用户主动上传的图片或视频文件里的内容，用户上传一张图片、多张图片或一个视频文件，后端调用检测服务处理完成后返回标注图与目标统计，整个流程是一次性的、被动的、离线的。最终目标网站要接入真实的摄像头，实现 24 小时不间断的实时监控，自动检测画面中的火焰和烟雾，发现异常立即生成预警并推送给相关人员，整个流程是持续的、主动的、在线的。两者的核心区别在于：前者是用户主动触发的离线检测工具，后者是系统主动监控的在线预警平台；前者的输入是文件，后者的输入是视频流；前者的输出是检测结果，后者的输出是预警与处置建议；前者的延迟可以接受秒级，后者的延迟必须控制在亚秒级；前者一次检测一次结束，后者需要长期稳定运行、断线重连、服务重启恢复。

### 四之二 从离线检测网站到在线预警网站需要添加的内容

　　一个只能检测对应图片或对应视频里的东西的网站，和一个可以真正接入仓库、实现预警功能的网站相比，需要添加以下几方面的内容。第一，摄像头接入层，从用户给一个摄像头到项目接入摄像头，需要支持 RTSP、RTMP、WebRTC 等主流视频流协议，需要摄像头管理界面与配置接口，需要摄像头状态监控与断线重连。第二，实时流处理层，需要视频流帧抽取（按固定帧率或跳帧策略）、批量推理（多帧合并提交以提高 GPU 利用率）、检测结果实时叠加（在原画面上绘制检测框并返回标注流）、延迟控制（端到端延迟控制在可接受范围）。第三，预警内容生成层，从检测结果到预警内容的转换，包括火情等级判定（基于火焰面积、烟雾面积、目标数量、置信度综合判定）、预警文案生成（根据场景与等级生成不同模板的预警文案）、预警推送渠道（站内消息、微信、邮件、短信、电话外呼等）。

　　第四，多场景支持层，仓库火灾预警、光伏电站巡检、森林防火检测、工地安全检测四个场景的差异化配置，包括检测阈值（不同场景对烟雾敏感度不同）、预警规则（不同场景的火情等级判定标准不同）、巡检周期（光伏电站可能只需白天巡检，森林防火需 24 小时）。第五，预警历史与统计层，预警记录查询、预警趋势分析、预警分级统计、预警处理状态跟踪。第六，持久化与高可用层，摄像头长期接入的数据存储（摄像头配置、心跳记录、抽帧快照）、断线重连（自动检测断线并尝试重连）、服务重启恢复（服务重启后自动恢复所有摄像头的监控任务）。第七，智能体编排层，多智能体协同工作以支持自然语言交互，让用户可以通过对话方式查询预警历史、分析检测结果、获取消防知识问答。

### 四之三 摄像头接入详细分析

　　摄像头接入是从用户给一个摄像头到项目接入摄像头的完整链路，需要以下步骤。第一步，用户在前端摄像头管理页面填写摄像头信息，包括名称、地址（RTSP 或 RTMP 或 WebRTC URL）、协议类型、所属场景、用户名密码（可选，用于 RTSP 鉴权）。第二步，前端调用 POST 接口注册摄像头，后端将摄像头信息存入 cameras 表，状态标记为 offline。第三步，后端启动摄像头接入任务，使用 OpenCV 或 ffmpeg-python 或 aiortc 等库建立与摄像头的连接，开始拉流。第四步，后端按固定帧率（如每秒 2 帧）抽取视频帧，送入检测服务的视频帧检测接口，使用现有的 video_confirmation_registry 多帧确认机制避免误报。第五步，后端将检测结果通过 WebSocket 实时推送给前端，前端在摄像头预览组件上叠加检测框并显示。第六步，当检测到火情且通过多帧确认后，后端调用预警服务生成预警记录，并通过配置的渠道推送预警。第七步，当摄像头断线时，后端自动标记状态为 offline 并尝试重连，重连成功后恢复监控任务。

### 四之四 实时流处理详细分析

　　实时流处理需要解决三个问题。第一是帧抽取策略，固定帧率（如每秒 2 帧）简单但可能漏检快速变化的火情，关键帧抽取（基于画面变化幅度）能减少冗余但实现复杂，建议初版采用固定帧率加跳帧策略，与现有视频检测的 frame_skip 参数一致。第二是批量推理，单帧推理效率低，可以将多帧合并为一个批次提交给模型以提高 GPU 利用率，但会增加延迟，需要在吞吐量与延迟之间权衡。第三是检测结果实时叠加，前端通过 WebSocket 接收检测结果后在视频画面上绘制检测框，需要前端有高效的视频帧渲染能力（建议使用 canvas 或 video 标签加 SVG 覆盖层）。延迟控制目标是端到端延迟不超过 2 秒，包括帧抽取（500 毫秒）、推理（500 毫秒）、传输（200 毫秒）、渲染（300 毫秒）等环节。

### 四之五 预警内容生成详细分析

　　预警内容生成需要从检测结果到预警内容的完整转换链路。第一是火情等级判定，基于火焰面积占比、烟雾面积占比、火焰目标数量、烟雾目标数量、最高置信度等综合判定，等级分为 safe（无火情）、notice（疑似烟雾或火苗，需关注）、warning（明显火情，需立即核查）、danger（严重火情，需立即处置）四级。第二是预警文案生成，根据场景与等级生成不同模板的预警文案，例如仓库场景 danger 级别生成"仓库 X 监控点检测到严重火情，火焰面积占比 X%，请立即疏散人员并拨打 119，启动消防设备"，光伏场景 warning 级别生成"光伏电站 X 区域检测到异常热斑或明火，请立即派员现场核查"。第三是预警推送渠道，站内消息通过 WebSocket 实时推送给在线用户，邮件通过 SMTP 服务发送，短信通过短信网关 API 发送，微信通过企业微信或公众号接口推送，电话外呼通过语音 API 推送给关键人员。第四是预警去重，同一摄像头短时间内多次检测到火情不应重复推送，应采用滑动窗口去重策略。

### 四之六 多场景支持详细分析

　　仓库火灾预警场景的特殊性是空间封闭、易燃物多、火势蔓延快，需要的额外功能是高灵敏度检测（低阈值以早发现）、24 小时不间断监控、危险品仓库的特殊处置预案。光伏电站巡检场景的特殊性是面积大、设备多、人工巡检效率低，需要的额外功能是支持无人机航拍图像上传检测、热斑与明火双重检测、白天巡检为主。森林防火检测场景的特殊性是范围大、距离远、烟雾先于明火出现，需要的额外功能是大视野远距离检测、烟雾敏感度提高、火点定位与风速风向联动。工地安全检测场景的特殊性是易燃物多、动火作业多、人员密集，需要的额外功能是动火作业许可联动、人员密集区域重点监控、施工时段重点监控。预警内容生成场景是一个横切能力，需要根据不同场景的差异化配置生成不同的预警文案与处置建议，需要预警文案模板库与场景化适配逻辑。

### 四之七 预警历史与统计详细分析

　　预警历史与统计需要支持预警记录查询（按时间、场景、等级、摄像头、处理状态筛选）、预警趋势分析（按日、周、月统计预警数量变化）、预警分级统计（各级别预警占比）、预警处理状态跟踪（已处理、处理中、未处理）、预警确认操作（用户确认收到预警并标记处理结果）。当前 fire_alerts 表已存在基本字段，但缺少预警确认人、确认时间、确认结果、关联摄像头等字段，需要扩展。

### 四之八 持久化与高可用详细分析

　　摄像头长期接入的数据存储需要新增 cameras 表存储摄像头配置、camera_heartbeats 表存储心跳记录、camera_snapshots 表存储抽帧快照（可选）。断线重连需要后端定时任务检测摄像头状态，断线后按指数退避策略重连。服务重启恢复需要后端启动时自动从数据库加载所有 active 状态的摄像头并恢复监控任务，需要使用后台任务队列（如 Celery 或 APScheduler）管理摄像头任务，避免单进程崩溃影响所有摄像头。

### 六之二 智能体模块接口契约

　　智能体模块的接口契约规定如下。主调度智能体输入为用户消息（字符串）加会话历史（列表），输出为路由决策（结构化字典），格式为：

```python
{
    "route_to": "detection_agent" | "analysis_agent" | "qa_agent" | "direct_reply",
    "intent": "detect" | "analyze" | "qa" | "chitchat",
    "args": { ... },  # 提取的参数，如文件路径、检测类型
    "confidence": 0.0 ~ 1.0
}
```

　　检测智能体工具调用协议规定如下。工具名为字符串，参数为字典，返回结果为字典。检测工具的签名约定：

```python
# 单图检测工具
tool_name = "detect_single_image"
args = {"image_path": str, "scene_id": int, "conf_threshold": float}
result = {"task_id": int, "fire_level": str, "fire_object_count": int, "smoke_object_count": int, "annotated_url": str}

# 批量检测工具
tool_name = "detect_batch_images"
args = {"image_paths": list[str], "scene_id": int}
result = {"total": int, "tasks": [{"task_id": int, "fire_level": str, ...}]}

# 压缩包检测工具
tool_name = "detect_zip"
args = {"zip_path": str, "scene_id": int}
result = {"total": int, "tasks": [...]}
```

　　分析智能体接口规定：输入为检测结果（字典）加场景信息（字典），输出为分析报告（Markdown 字符串）加处置建议（字符串）加风险等级（字符串）。问答智能体接口规定：输入为问题（字符串）加检索结果（列表），输出为回答（Markdown 字符串）加引用来源（列表）。SSE 流式事件类型定义如下，所有事件使用 event 与 data 两个字段，data 为 JSON 字符串：

```
event: agent_start    data: {"agent": "supervisor", "intent": "detect"}
event: tool_call      data: {"tool": "detect_single_image", "args": {...}}
event: tool_result    data: {"tool": "detect_single_image", "result": {...}}
event: message_chunk  data: {"content": "片段文本"}
event: agent_end      data: {"agent": "supervisor", "tokens_used": int, "latency_ms": int}
event: error          data: {"message": "错误描述"}
```

| 方法 | 路径 | 参数 | 响应 |

| GET | /api/v1/alert-rules | query: scene_id? | {code:200, data: [...]} |

　　预警内容生成规则规定如下。火情等级判定基于火焰面积占比（fire_area）、烟雾面积占比（smoke_area）、火焰目标数量（fire_object_count）、烟雾目标数量（smoke_object_count），判定逻辑：

```
if fire_area >= 0.05 or fire_object_count >= 3: fire_level = "danger"
elif fire_area >= 0.02 or fire_object_count >= 1: fire_level = "warning"
elif smoke_area >= 0.05 or smoke_object_count >= 2: fire_level = "notice"
else: fire_level = "safe"
```

　　预警文案模板按场景与等级组合，仓库场景 danger 级别文案模板为："仓库 {scene_name} 监控点 {camera_name} 检测到严重火情，火焰面积占比 {fire_area_percent}%，请立即疏散人员并拨打 119，启动消防设备"。光伏场景、森林场景、工地场景各有对应模板，由 content_builder 模块管理。

### 七之一 总体协作原则

　　后端在新增或修改接口前，先确定接口路径、请求方法、参数名和响应字段，并同步给前端和项目经理。前端按后端接口实现，若发现字段名或返回结构与后端不一致，优先与后端协商统一，不自行新增字段或改变字段含义。所有新增接口统一使用 /api/v1 前缀（保留接口除外），返回格式沿用已有的 {code, message, data} 统一结构。流式对话接口沿用现有路径 /api/chat/messages/stream，具体事件类型和字段名由后端定义，但需在合并前文档化并告知前端。

### 七之二 已稳定的接口契约

### 七之三 新增接口与字段约定

## 八、提交规范与审核规则

### 八之一 提交内容必须与主干接口完全适配

　　提交内容必须与主干接口完全适配，必须修复 bug，必须做测试，必须听取 AI 的改进建议。目前很多人提交的内容完全无法合并到主干，只有部分内容可取，这种现象不能再出现。具体要求如下：第一，提交前必须实际运行测试，后端运行 pytest，前端运行 npm run build，服务启动后进行网页操作验证，测试通过才能提交。第二，提交内容必须基于当前主分支最新版本，避免基线过旧导致大面积冲突。第三，提交内容必须修复已知的 bug，不能带着已知 bug 提交。第四，提交内容必须听取 AI 的改进建议，对 AI 审核评论中提出的问题必须逐条响应，不能视而不见。

### 八之二 前端提交内容依赖后端但后端未做时的判定规则

　　管理员审核仓库时，如果一个前端提交的内容依赖一些后端内容，但后端目前没有做或没有上传仓库，只要这个前端还能跑起来，即使对应的后端内容没有做，也不算它出了严重 bug。因为它是在往现有的工程上增加内容，增加了和原本不冲突的内容，那我们就认为它是可以接受的。但如果它是在往现有的仓库内容上添加和现有内容冲突的内容，或者添加的内容会导致整个项目出问题，那我们就认为它是不可接受的。具体判定规则如下：第一，前端新增页面或组件不依赖后端新接口时，只要前端能编译通过、能正常访问、不破坏现有页面，即可接受。第二，前端新增页面或组件依赖后端新接口但后端未实现时，前端应使用 mock 数据或占位实现保证页面能跑起来，并在代码中明确标注 TODO 等待后端实现，即可接受。第三，前端修改现有页面或组件时，必须保持与现有接口兼容，不能修改现有接口的字段名或含义，不能删除现有功能，否则不可接受。第四，前端提交内容导致 npm run build 失败、导致现有页面无法访问、导致整个项目启动失败的，不可接受。

### 八之三 接口不适配处理

　　接口不适配分为硬不适配与软不匹配两类。硬不适配包括：接口签名错（路径、方法、参数名、响应字段不一致）、编译失败（前端 npm run build 失败、后端导入失败）、运行时崩溃（服务启动后访问接口报 500 错误）、核心功能缺失（接口存在但核心逻辑未实现）。硬不适配在审核阶段打回，不合并，作者修复后重新提交。软不匹配包括：接口能跑通但未充分利用已有功能（如重复实现已有服务）、未对齐约定（如未使用统一响应格式、未使用统一前缀）。软不匹配合并后提醒，作者在后续提交中改进。

### 八之四 修 bug 与测试要求

　　提交前必须实际运行测试，测试通过才能提交。后端测试要求：运行 pytest 命令，所有测试用例通过，无失败、无错误。前端测试要求：运行 npm run build 命令，构建成功无报错；运行 npm run dev 命令，服务能正常启动；访问主要页面（登录、检测、历史、看板、训练、对话）无白屏、无控制台错误。服务启动测试要求：使用 docker-compose up 命令一键启动全部服务，所有服务健康检查通过；访问后端 /docs 接口文档能正常加载；访问前端首页能正常加载。网页操作测试要求：登录、上传图片检测、查看历史记录、查看数据看板、启动训练任务、发送对话消息等核心流程能正常完成。测试不通过的提交不得推送，已推送的由管理员打回。

### 八之五 最多改 3 次规则

　　未通过审核的提交，按 AI 要求修改，顶多改 3 次，一定可以通过审核。超过 3 次仍未通过，说明提交内容存在根本性问题，需要重新设计。具体规则如下：第一次审核未通过，管理员在 PR 评论中给出修改意见，作者按意见修改后重新提交；第二次审核未通过，管理员再次给出修改意见，作者按意见修改后重新提交；第三次审核未通过，管理员关闭该 PR，作者需要重新设计提交内容，从新 PR 开始。每次修改必须在 PR 评论中回复已修改并说明修改内容，管理员收到回复后再次审核。

### 八之六 前端界面美化要求

　　明确要求前端人员优化目前美观的界面使其更美观。在完成功能基础上提升整体视觉质量，包括：配色方案（统一主色调与辅助色，避免色彩杂乱）、字体规范（统一字体族与字号层级，标题与正文区分清晰）、间距留白（统一间距 token，避免过密或过疏）、过渡动画（合理使用过渡动画提升交互流畅度，避免过度动画）、加载状态（所有异步操作必须有加载状态指示，避免用户以为卡死）、表单表格视觉（表单输入框与表格样式统一，对齐规范）、暗色与亮色模式一致性（若支持暗色模式，必须保证两种模式下均可读）、响应式适配（适配主流屏幕尺寸，至少支持 1280 到 1920 宽度）。如果前端做的东西和后端东西不匹配，就主动让后端的人去改，不要在那里什么都不干。前端不能以"后端没做"为借口不推进，应主动推动后端实现，或使用 mock 数据先推进前端。

### 八之七 测试人员克隆仓库到本地测试反馈

　　明确要求测试人员克隆仓库到本地、运行、测试、通过微信反馈 bug。具体要求如下：测试人员将目前这个仓库克隆到本地来，然后开始运行，进行各种测试。把测试出来的目前使用网站的 bug，比如点开哪个界面会出问题，或者用哪个功能会导致整个工程崩溃，都找出来。找出来之后，在微信上反馈给相关部分的人，让他去修。测试人员需要覆盖的测试场景包括：登录注册流程、单图检测流程、批量检测流程、视频检测流程、历史记录查询流程、数据看板查看流程、训练任务启动流程、对话消息发送流程、所有页面的访问与交互、不同浏览器（Chrome、Edge）的兼容性、不同屏幕尺寸的响应式。测试人员需要记录 bug 的复现步骤、预期结果、实际结果、严重等级，便于开发人员修复。

## 十一、注意事项与风险提示

### 十一之一 大模型占位符约束

　　大模型接口默认采用占位符实现，主要原因是当前经费不足，避免产生额外调用费用。任何引入真实大模型接口调用的提交，只要作者愿意自费承担相应费用、不泄露密钥且不影响现有功能，可以支持接入，但仍需通过合并请求审核。请各角色在开发过程中特别注意，不要默认接入需要付费的真实接口。环境变量 LLM_STUB_MODE=true 必须保持，所有大模型调用必须检查该环境变量，为 true 时使用占位实现，为 false 时才允许真实调用。算法方向同学在 PR #8 中的真实调用问题必须在本次优先修复，作为后续提交的参照样板。

### 十一之二 主分支保护规则

　　主分支已经配置了保护规则，所有代码改动必须通过合并请求提交，并经过至少一人审核后才能合并。禁止直接推送到主分支，也禁止强制推送。禁止删除已合并到主分支的文件，违反以先提交内容为准铁律。提交前必须基于当前主分支最新版本基线化，避免基线过旧导致大面积冲突。已合并的祖先分支已打上 old-version 标签，后续合并工作不再考虑这些分支，避免重复劳动。

### 十一之三 智能体联调依赖接口契约

　　智能体联调依赖于前后端接口契约，建议后端工程师尽早确定主调度路由协议、流式事件格式和工具调用参数结构，并与前端工程师同步。前端同学本地进度与主干差距较大，更需要注意接口对齐，避免合并时大面积冲突。SSE 流式事件类型的扩展（agent_start、tool_call、tool_result、message_chunk、agent_end）必须先文档化并告知前端，前端才能正确实现流式渲染。

### 十一之四 前端自由度与进度评估

　　前端同学的对话页面重写和界面美化工作给予较大自由度，不强制本阶段全部完成，但需要给出明确的当前进度和剩余工作量评估，便于项目经理统筹。前端同学应优先保证 PR #8 修复合并，再推进新页面与美化工作。前端不能以"后端没做"为借口不推进，应主动推动后端实现，或使用 mock 数据先推进前端。如果前端做的东西和后端东西不匹配，应主动让后端去改，不要在那里什么都不干。

### 十一之五 被阻断提交计入进度但需优先修复

　　被阻断或存在问题的提交计入工作进度中的完成但存在问题状态，相关作者需要优先修复阻断问题，避免影响整体联调进度。PR #8 的 3 个 P0 问题、PR #14 与 PR #15 的冲突、PR #17 的待审核状态，均需在本阶段优先处理。最多改 3 次规则严格执行，超过 3 次未通过的 PR 予以关闭，作者需重新设计提交内容。

### 十一之六 详细代码框架为建议性但功能目标需实现

　　本文档第六章的详细代码框架仅为建议性框架，各角色可以根据实际情况调整目录结构、文件命名、内部实现，但文档中列出的功能目标需要实现。接口契约（路径、方法、参数、响应格式）必须严格遵守，避免出现接口不适配。数据库表与字段约定必须通过 Alembic 迁移脚本管理，禁止手动修改数据库结构。鼓励在完成最低标准后补充更多便利功能，使平台更易用。

### 十一之七 时间紧张需把握优先级

　　时间较为紧张，各角色应把握优先级。修复阻断问题优先于新功能开发，核心链路打通优先于边缘功能完善。多智能体骨架与摄像头实时检测链路是核心路径，应优先推进；预警生成与推送、多场景配置是重要路径，其次推进；知识库搭建、界面美化、模型评估与导出完善是辅助路径，最后推进。如果进度落后，应及时向项目经理汇报，协商调整范围，不要默默延期。各角色之间应主动沟通，避免出现"前端等后端、后端等前端"的相互等待局面。测试人员应及时反馈 bug，开发人员应及时修复 bug，形成良性循环。

### 十一之八 端口与部署风险

　　docker-compose.yml 中后端服务映射主机端口 8000，前端服务映射主机端口 80，如果主机已有其他服务占用这些端口会导致容器启动失败。部署前需检查端口占用情况，必要时修改 docker-compose.yml 中的端口映射。生产环境的 JWT 密钥、数据库密码、MinIO 凭据必须修改为强密码，禁止使用默认值。模型权重文件较大，不应提交到仓库，应通过对象存储或单独的下载渠道分发。
