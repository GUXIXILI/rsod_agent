# Day1-Day10 讲义补齐指导计划

## 一、当前验收状态与欠缺清单

　　本项目截至当前已完成 Day1 至 Day9 讲义的综合验收工作，整体结论为基本通过，但仍存在 7 个待修复问题，其中 P0 级别 2 个、P1 级别 2 个、P2 级别 4 个。Day10 讲义涉及的页面与功能尚未逐项验收，需要各角色按本指引补齐后统一交付验收。以下逐条列明 7 个问题的编号、问题描述、位置、优先级、责任角色与对应讲义天数，作为后续修复与验收的依据。
　　P0-1 摄像头 WebSocket 后端未实现。问题位置在 backend 缺少 ws 端点，前端已用 ws 协议连接但后端无对应路由，优先级为 P0，责任角色为后端工作人员，对应 Day09 讲义。
　　P0-2 LangChain ReAct Agent 未实现。问题位置在 backend/app/services/chat_service.py，当前仍为占位回复，优先级为 P0，责任角色为 AI 算法工作人员，对应 Day08 讲义。
　　P1-1 detectVideo 导入被注释。问题位置在 frontend/src/views/DetectionPage.vue 第 93 行，导致视频检测功能无法运行，优先级为 P1，责任角色为前端工作人员，对应 Day09 讲义。
　　P1-2 detect_zip 后端缺失。问题位置在 backend/app/api/detection.py，缺少压缩包批量检测接口与对应 service 方法，优先级为 P1，责任角色为后端工作人员，对应 Day08 讲义。
　　P2-1 训练 DELETE 接口缺失。问题位置在 backend/app/api/training.py，无法删除训练任务，优先级为 P2，责任角色为后端工作人员，对应 Day06 讲义。
　　P2-2 TrainingService 缺少 validate_model 与 export_model 方法。问题位置在 backend/app/training/training_service.py，service 层职责不完整，优先级为 P2，责任角色为后端工作人员配合 AI 算法，对应 Day07 讲义。
　　P2-3 通用 API 限流未实现。问题位置在 backend 无通用限流中间件，优先级为 P2，责任角色为后端工作人员，对应 Day04 讲义。
　　P2-4 Dockerfile libgl1-mesa-glx 不可用。问题位置在 Dockerfile 第 9 行，Debian Trixie 源中该包已不存在，优先级为 P2，责任角色为后端工作人员，对应 Day01 讲义。
　　Day10 待验收项尚未逐项验收，需要补齐的内容包括以下三类。第一类是数据看板页面，需要补齐总览统计卡片、火情分布饼图、近 7 天趋势折线图三类图表。第二类是历史记录页面，需要补齐分页列表、按时间场景火情等级筛选、详情查看、单条与批量删除功能。第三类是功能完善，包括双版本共存择优、模型评估展示、导出下载入口，以及豆包式多 agent 统一入口页面作为长期目标。

## 二、项目经理职责指引

　　项目经理作为整体推进的核心协调者，需要对照实训计划与需求文档校准各角色任务优先级，确保 P0 问题优先修复、P1 问题紧跟其后、P2 问题在不阻塞主线的前提下并行推进。
　　每日需组织固定时间召开站会，每个角色简报昨日完成内容、今日计划任务与当前阻塞点，会后形成会议纪要存档。
　　当前端与后端接口字段不匹配或联调出现阻塞时，项目经理需主动协调对接会，拉齐双方对接口契约的认知。
　　里程碑跟踪方面，需对照本指引第十部分的六个阶段里程碑确认进度，发现偏差及时调整资源。
　　风险预警方面，发现某角色连续 2 天未完成预期任务时需主动介入，了解原因并协调支援。
　　每日 17:00 前需收集各角色当日进度，更新到 d 盘 clone 目录下工作进度目录中的进度文档。
　　对外沟通方面，项目经理负责与指导老师、需求方对接需求变更与验收安排。

## 三、GitHub 仓库管理员职责指引

　　仓库管理员负责主干分支保护与 PR 审核合并。PR 审核遵循 P0、P1、P2 分级策略，P0 问题阻塞合并直到修复完成，大多数 P1 问题同样阻塞合并，P2 问题仅作提醒不阻塞。
　　合并策略遵循只增不减原则，纯增量内容直接吸收，同名页面冲突时新开独立页面实现双版本共存，纯删除与配置替换不吸收。
　　合并方式统一使用 Create a merge commit 即保留完整提交历史的方式，不走 squash 与 rebase。
　　主干保护规则方面，禁止直接 push 到 main 分支，必须通过 PR 流程，且至少获得 1 个 review 才能合并。
　　合并前需确认提交者是否在 PR 描述中标注提醒目标，即负责某某角色的人，以便后续联调跟进。
　　对完全无价值的分支需及时关闭删除，参考 7 月 15 日已关闭 5 个重复分支的操作。
　　对有价值的分支需督促作者修复 P0 问题后重新提交。
　　审核完成后需清理临时测试脚本与测试数据库文件，确保工程目录整洁。

## 五、后端工作人员职责指引

　　后端工作人员按优先级从高到低推进以下任务。最高优先级是修复 P0-1 问题，即实现 WebSocket 路由 /api/detection/camera，处理摄像头帧接收、YOLO 推理、标注帧返回，前端已用 ws 协议连接，后端需对接。
　　第二优先级是修复 P1-2 问题，即添加 POST /api/detection/zip 接口与 DetectionService.detect_zip 方法，处理压缩包解压批量检测。
　　第三优先级是修复 P2-1 问题，即添加 DELETE /api/training/{task_id} 接口，支持训练任务删除。
　　第四优先级是修复 P2-2 问题，即将 validate_model 与 export_model 占位实现从 API 层移入 TrainingService，保持 service 层职责完整。
　　第五优先级是修复 P2-3 问题，即实现基于 Redis 的通用 API 限流中间件，覆盖所有写接口。
　　第六优先级是修复 P2-4 问题，即将 Dockerfile 第 9 行 libgl1-mesa-glx 替换为 libgl1 或 libgl1-mesa-dri，兼容 Debian Trixie。

## 六、AI 算法工作人员职责指引

　　AI 算法工作人员按优先级从高到低推进以下任务。最高优先级是修复 P0-2 问题，即集成 LangChain ReAct Agent 替换 chat_service 占位回复，需安装 langchain 与 langchain-community 依赖。
　　第二优先级是将检测能力封装为 Tool，包括 detect_single、detect_batch、detect_video，封装为 LangChain Tool 供 Agent 调用。
　　第三优先级是 ONNX 导出能力验证，配合 P2-2 修复，验证 model_artifact_service 的 ONNX 导出实际可用。
　　第四优先级是数据集生成，运行 backend/tools/prepare_fire_smoke_dataset.py 生成 datasets/fire_smoke/ 目录，补齐 Day5 欠缺内容。
　　第五优先级是保持 LLM_STUB_MODE 占位可切换，LLM_STUB_MODE 等于 true 时占位实现可用，切换为 false 时可接入真实 API。
　　第六优先级是模型评估脚本完善，evaluate_fire_smoke_model.py 已存在，需在实际验证集上运行确认指标合理。
　　第七优先级是模型训练优化，当前 fire_smoke_yolo11n_v1 模型 Fire 类 Recall 偏低为 0.610，需补充仓库、室内、夜间、强光场景数据或升级 YOLO11s。

## 七、测试人员职责指引

　　测试人员负责全栈环境搭建与端到端验证。首先克隆仓库到本地，执行 git clone https://github.com/GUXIXILI/rsod_agent.git。
　　其次使用 docker compose 一键启动，执行 docker compose up -d --build，确认 PostgreSQL、Redis、MinIO、后端、前端 5 个容器都 Up 且 healthy。
　　重点验证 7 个修复问题对应的端到端流程。验证 P0-1 时打开 /camera-detection 页面，点击开启摄像头，确认 WebSocket 连接成功并显示检测结果。验证 P0-2 时在对话页面输入帮我检测这张图片有没有火灾，确认 Agent 调用检测 Tool 而非占位回复。验证 P1-1 时在 /detection 页面上传视频，确认视频检测可运行不报 ReferenceError。验证 P1-2 时在 /detection 页面上传 zip 压缩包，确认批量检测可运行。验证 P2-1 时在训练页面删除训练任务，确认任务被删除。验证 P2-2 时在训练页面点击评估与导出，确认 ONNX 文件可下载。验证 P2-3 时高频调用写接口，确认限流中间件返回 429。验证 P2-4 时确认 docker compose build 不再报 libgl1-mesa-glx 错误。
　　Day10 页面覆盖方面，需确认数据看板三类图表渲染正常、历史记录分页筛选详情删除都可用。
　　发现 bug 时通过微信反馈，需包含复现步骤、预期结果、实际结果、截图。
　　pytest 全量回归方面，进入 backend 目录执行 pytest，确认 188 以上用例全部通过。
　　Vitest 全量回归方面，进入 frontend 目录执行 npm run test，确认 5 以上用例全部通过。

## 八、7 个问题修复方案

　　P0-1 摄像头 WebSocket 后端未实现。责任角色为后端工作人员。修复步骤如下，第一步在 backend/app/api/detection.py 或新建 ws 路由文件中注册 /api/detection/camera 的 WebSocket 端点，第二步在端点内接收前端发送的视频帧，第三步调用 YOLO 推理服务对帧进行检测与标注，第四步将标注后的帧以 base64 或二进制形式返回前端，第五步处理连接异常与心跳保活。验证方式为打开 /camera-detection 页面点击开启摄像头，确认画面实时显示检测结果。预期完成时间窗口为 1 到 2 天。
　　P0-2 LangChain ReAct Agent 未实现。责任角色为 AI 算法工作人员。修复步骤如下，第一步安装 langchain 与 langchain-community 依赖，第二步将 detect_single、detect_batch、detect_video 封装为 LangChain Tool，第三步在 chat_service.py 中初始化 ReAct Agent 并注入检测 Tool，第四步将原有占位回复替换为 Agent 的 invoke 调用，第五步在 LLM_STUB_MODE 为 true 时保留占位逻辑以便离线测试。验证方式为在对话页面输入帮我检测这张图片有没有火灾，确认 Agent 调用检测 Tool 而非返回占位文本。预期完成时间窗口为 1 到 2 天。
　　P1-1 detectVideo 导入被注释。责任角色为前端工作人员。修复步骤如下，第一步打开 frontend/src/views/DetectionPage.vue 第 93 行，第二步取消 detectVideo 导入语句的注释，第三步确认导入路径与导出名称一致，第四步在 /detection 页面上传视频确认可运行。验证方式为上传视频后不报 ReferenceError 且检测正常执行。预期完成时间窗口为 1 天。
　　P1-2 detect_zip 后端缺失。责任角色为后端工作人员。修复步骤如下，第一步在 backend/app/api/detection.py 添加 POST /api/detection/zip 接口，第二步在 DetectionService 中实现 detect_zip 方法，第三步在方法内处理压缩包解压与批量图片遍历，第四步对每张图片调用 detect_single 并汇总结果，第五步返回包含每图检测框与统计摘要的响应。验证方式为在 /detection 页面上传 zip 压缩包，确认批量检测返回完整结果。预期完成时间窗口为 1 天。
　　P2-1 训练 DELETE 接口缺失。责任角色为后端工作人员。修复步骤如下，第一步在 backend/app/api/training.py 添加 DELETE /api/training/{task_id} 接口，第二步在 TrainingService 中实现删除逻辑，第三步清理训练任务关联的临时文件与数据库记录，第四步返回删除结果。验证方式为在训练页面删除训练任务，确认任务从列表消失且数据库记录已清除。预期完成时间窗口为 2 到 3 天。
　　P2-2 TrainingService 缺少 validate_model 与 export_model 方法。责任角色为后端工作人员配合 AI 算法。修复步骤如下，第一步将 validate_model 与 export_model 占位实现从 API 层移入 TrainingService，第二步在 validate_model 中调用评估脚本计算指标，第三步在 export_model 中调用 model_artifact_service 导出 ONNX，第四步保持 API 层仅做参数校验与调用转发。验证方式为在训练页面点击评估与导出，确认评估报告显示且 ONNX 文件可下载。预期完成时间窗口为 2 到 3 天。
　　P2-3 通用 API 限流未实现。责任角色为后端工作人员。修复步骤如下，第一步引入基于 Redis 的限流中间件，第二步为每个写接口配置默认限流阈值，第三步在中间件中统计单位时间请求次数，第四步超阈值时返回 429 状态码与友好提示。验证方式为高频调用写接口，确认限流中间件返回 429。预期完成时间窗口为 2 到 3 天。
　　P2-4 Dockerfile libgl1-mesa-glx 不可用。责任角色为后端工作人员。修复步骤如下，第一步打开 Dockerfile 第 9 行，第二步将 libgl1-mesa-glx 替换为 libgl1 或 libgl1-mesa-dri 以兼容 Debian Trixie，第三步重新执行 docker compose build 确认镜像构建成功。验证方式为 docker compose build 不再报 libgl1-mesa-glx 找不到的错误，且后续容器正常运行。预期完成时间窗口为 2 到 3 天，实际可半天内完成。

## 九、Day10 补齐计划

　　Day10 讲义标题为基于 YOLO11n 的目标检测智能体平台数据看板加历史记录加功能完善，分数据看板、历史记录、功能完善三个维度。
　　数据看板维度需补齐总览统计卡片，包括总检测数、今日检测数、火灾告警数、模型数四项指标；需补齐火情分布饼图，展示 safe、notice、warning、danger 四级占比；需补齐趋势折线图，展示近 7 天检测数与火灾数变化。该维度责任角色为前端工作人员负责页面、后端工作人员负责统计接口。
　　历史记录维度需补齐分页列表，每页 20 条；需补齐筛选功能，支持按时间范围、场景、火情等级筛选；需补齐详情查看，点击单条记录展开详情；需补齐删除功能，支持单条删除与批量删除。该维度责任角色为前端工作人员负责页面、后端工作人员负责接口。
　　功能完善维度需补齐双版本共存择优，即 DetectionPage 与 CameraDetectionPage 在实际使用中比较择优；需完善模型评估展示页面；需完善导出下载入口；豆包式多 agent 统一入口页面为可选的长期目标。该维度责任角色为前端工作人员与 AI 算法工作人员。
　　依赖关系方面，数据看板依赖统计接口已存在，Day4 已有 /api/stats 接口可直接复用或扩展；历史记录依赖 Day2 数据库模型已存在，DetectionResult 表可直接复用。两个维度的前置依赖均已就绪，可并行推进。

## 十一、注意事项与风险提示

　　第一，LLM_STUB_MODE 等于 true 约束下大模型调用必须使用占位实现，不主动发起真实远程调用，避免触发外部 API 计费或合规风险。
　　第二，前端依赖后端但后端未做时，前端能跑起来不算严重 bug，与现有内容冲突或导致项目出问题才算不可接受。
　　第三，双版本共存页面即 DetectionPage 与 CameraDetectionPage 需在实际使用中比较择优，不要急于删减。
　　第四，测试时已结束的测试及时关闭，控制内存占用避免电脑死机。
　　第五，使用 trae 内置浏览器测试，不使用电脑浏览器软件。
　　第六，不安装未经许可的软件。
　　第七，提交前先拉取最新主干，在自己分支上解决冲突。
　　第八，PR 描述要写清楚改了什么、为什么改、测试过没有、测试结果是什么。
　　第九，不要提交连自己都没跑过的代码。
　　第十，不要把大模型真实 API 调用塞进占位符位置。
