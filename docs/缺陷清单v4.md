### BUG-001: Agent对话LLM报400（function.arguments为None）

| 项目 | 内容 |
|------|------|
| **编号** | BUG-001 |
| **优先级** | P0 |
| **严重程度** | 严重 |
| **模块** | Agent对话 / LLM集成 |
| **发现轮次** | 第1轮（最初表现为502，第4轮确认为400） |
| **状态** | 🔴 未修复 |

**问题描述**：
Agent调用工具后，LLM返回400参数验证错误，function.arguments字段为None而非有效JSON字符串。

**错误现象**：
- 前端：AI回复中断，显示"Agent执行出错"
- SSE事件序列：thinking → start → tool_start → tool_call → tool_end → tool_result → error → done
- 错误信息：
```
Error code: 400 - function.arguments 
Input should be a valid string [type=string_type, input_value=None, input_type=NoneType]
```

**推断原因**：
1. Agent在构造第二轮LLM请求时，将tool_call的arguments设为了None
2. 工具返回结果后，重新组装消息历史时function.arguments字段丢失
3. LangGraph或自定义Agent代码在处理tool_end事件时格式不正确

**影响范围**：
- Agent对话功能不可用（工具调用后无法继续生成回复）
- RAG增强对话不可用
- 多轮对话无法正常进行

**验证方式**：
1. 进入智能对话页面
2. 发送"系统有多少用户？"触发工具调用
3. 观察SSE流中最终的error事件
4. 检查错误信息中是否包含function.arguments相关内容

**建议修复**：
1. 检查Agent组装tool消息时function.arguments字段是否正确传递
2. 确保tool_call消息的arguments始终是字符串类型（JSON序列化）
3. 检查LangGraph节点之间的状态传递是否正确保留了工具调用参数
4. 添加单元测试验证工具调用的完整消息链格式

### BUG-009: 视频检测任务失败且无错误日志

| 项目 | 内容 |
|------|------|
| **编号** | BUG-009 |
| **优先级** | P0 |
| **严重程度** | 严重 |
| **模块** | 视频检测 / 异步任务 |
| **发现轮次** | 第3轮（最初表现为MinIO 431，第4轮确认性质变化） |
| **状态** | 🔴 未修复 |

**问题描述**：
上传视频后，视频检测异步任务失败，数据库状态为failed，但后端日志中没有对应的ERROR日志记录。状态查询API返回unknown而非failed。

**错误现象**：
- 视频上传成功，返回task_id
- 进度始终卡在10%
- 状态API返回：status=unknown, progress=10
- 数据库实际状态：failed（从历史记录详情接口确认）
- 后端日志：本轮测试无新增ERROR日志

**推断原因**：
1. 视频检测任务中的异常被静默捕获（try-except后未记录日志）
2. 异步任务线程异常退出，未正确更新任务状态
3. 状态查询API读取的是Redis缓存而非数据库，缓存未同步失败状态
4. 早期的431错误可能已修复，但引入了新的失败原因

**影响范围**：
- 视频检测功能完全不可用
- 用户无法得知任务失败原因
- 错误排查困难（无日志可查）

**验证方式**：
1. 上传MP4视频文件
2. 轮询任务状态API，观察是否始终为unknown
3. 调用历史记录详情接口，确认数据库中实际状态为failed
4. 检查后端日志是否有对应的ERROR记录

**建议修复**：
1. 检查视频检测异步任务的异常处理逻辑，确保所有异常都记录ERROR日志
2. 修复状态查询API，使其正确读取数据库中的failed状态（BUG-013）
3. 在任务失败时，将失败原因写入数据库的results或error_message字段
4. 考虑添加任务超时机制，避免任务永久卡在processing状态

### BUG-013: 视频状态查询API返回unknown，数据库实际为failed

| 项目 | 内容 |
|------|------|
| **编号** | BUG-013 |
| **优先级** | P1 |
| **严重程度** | 高 |
| **模块** | 视频检测 / API |
| **发现轮次** | 第4轮 |
| **状态** | 🔴 未修复 |

**问题描述**：
视频状态查询接口`GET /api/detection/video/status/{id}`返回status=unknown，但数据库中任务的实际状态为failed。

**错误现象**：
- 状态API返回：`{"status": "unknown", "progress": 10}`
- 历史记录详情API返回：`{"status": "failed"}`
- 两个接口对同一任务返回不同状态

**推断原因**：
1. 状态API优先从Redis缓存读取状态，缓存未更新为failed
2. 任务失败时只更新了数据库，未更新Redis缓存
3. 状态API的状态映射逻辑有问题，failed状态被映射为unknown

**影响范围**：
- 用户无法正确得知视频检测任务已失败
- 用户可能误以为任务仍在处理中，长时间等待
- 前端进度条永远卡在10%

**验证方式**：
1. 上传视频触发检测任务
2. 等待足够长时间（如30秒以上）
3. 调用`GET /api/detection/video/status/{id}`，观察返回的status
4. 调用`GET /api/history/tasks/{id}`，对比返回的status
5. 两者不一致即确认此BUG

**建议修复**：
1. 检查video status API的实现，确认其状态来源（Redis vs 数据库）
2. 确保任务失败时同时更新数据库和Redis缓存
3. 增加状态映射的完整性，确保failed状态被正确识别和返回
4. 考虑在unknown状态持续一段时间后自动从数据库刷新状态

### BUG-008: Element Plus Radio弃用警告

| 项目 | 内容 |
|------|------|
| **编号** | BUG-008 |
| **优先级** | P2 |
| **严重程度** | 低 |
| **模块** | 前端组件 |
| **发现轮次** | 第2轮 |
| **状态** | 🔴 未修复 |

**问题描述**：
使用el-radio组件时控制台出现弃用警告，label属性作为value的用法将在3.0.0版本移除。

**错误现象**：
- 浏览器控制台警告：
```
[warn] ElementPlusError: [el-radio] [API] label act as value is about to be 
deprecated in version 3.0.0, please use value instead.
```

**推断原因**：
1. 项目使用的Element Plus版本较新，引入了新的value属性
2. 代码中仍使用旧版的label属性来设置单选框值

**影响范围**：
- 不影响功能，但控制台有警告
- 未来升级到Element Plus 3.0可能导致功能失效

**验证方式**：
1. 进入摄像头检测页面
2. 打开浏览器控制台
3. 观察是否有Radio弃用警告

**建议修复**：
1. 将el-radio的label属性改为value属性
2. 全局搜索所有el-radio用法并统一修改
3. 升级Element Plus后进行回归测试

|------|------|
| **编号** | BUG-012 |
| **优先级** | P3 |

| **发现轮次** | 第4轮 |
| **状态** | 🔴 待确认 |

**问题描述**：
角色列表API只返回1个角色，可能缺少完整的角色权限体系。

**错误现象**：
- GET /api/roles 只返回1条记录
- 可能只有admin角色，缺少普通用户、访客等角色

**推断原因**：
1. 角色表中只初始化了admin角色
2. 角色权限系统尚未完全实现
3. 可能是设计如此，单角色系统

**影响范围**：
- 影响用户权限管理功能

**验证方式**：
1. 调用GET /api/roles接口
2. 检查返回的角色数量和列表
3. 查看讲义中是否有多角色要求

**建议修复**：
1. 根据讲义要求确认是否需要多角色
2. 如需多角色，初始化admin/user/guest等角色
3. 实现基于角色的权限控制（RBAC）
