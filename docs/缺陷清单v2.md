## 二、未修复缺陷详情

### BUG-001：Agent对话执行失败，LLM返回400错误

| 字段 | 内容 |
|------|------|
| **缺陷编号** | BUG-001 |
| **优先级** | P0（必须立即修复） |
| **所属模块** | Day11 - Agent智能对话 |
| **影响范围** | 所有需要工具调用的对话功能 |

#### 缺陷现象
在智能对话页面发送需要调用工具的消息时，AI回复失败，页面显示"Agent 执行出错"。

错误信息：
```
Error code: 400 - 3 validation errors for ValidatorIterator
0. ChatCompletionMessageFunctionToolCallParam.function.arguments
   Input should be a valid string
```

#### 复现步骤
1. 登录系统，进入智能对话页面
2. 输入"系统有多少用户？"或其他会触发工具调用的问题
3. 点击发送
4. 观察AI回复结果

#### 错误原因推断
LangGraph中工具节点返回结果时，生成的 tool call 消息的 `function.arguments` 字段为 `None`（空值），而 LLM API 要求该字段必须是有效的 JSON 字符串。

可能的问题点：
- 工具调用结果格式化逻辑有问题
- tool call message 构建时遗漏了 arguments 字段
- 工具返回的内容没有正确序列化为JSON字符串

#### 后端日志验证
从控制台错误和SSE响应来看，错误发生在LLM调用阶段，工具本身执行成功了（返回了用户列表数据），但在将工具结果传回LLM继续推理时，消息格式不符合LLM API要求。

### BUG-004：个人设置页面不显示个人信息

| 字段 | 内容 |
|------|------|
| **缺陷编号** | BUG-004 |
| **优先级** | P1（尽快修复） |
| **所属模块** | Day10 - 个人设置 |
| **影响范围** | 用户查看个人信息功能 |

#### 缺陷现象
进入个人设置页面后，左侧个人信息卡片显示"暂无个人信息"或空白，无法看到用户名、邮箱、角色等基本信息。

#### 复现步骤
1. 登录系统
2. 点击顶部导航"个人设置"
3. 观察左侧个人信息卡片

#### API验证
后端API `GET /api/user/profile` 返回数据正常：
```json
{
  "code": 200,
  "message": "success",
  "data": {
    "id": 32,
    "username": "admin",
    "email": "test_duplicate@example.com",
    "phone": null,
    "avatar": null,
    "is_active": true,
    "roles": ["admin"],
    "created_at": "2026-07-08T18:48:10.236676"
  }
}
```

#### 错误原因推断
前端 `SettingsPage.vue` 中数据处理逻辑可能存在问题：
- 响应数据解包逻辑错误（判断条件 `res.data.username !== undefined` 可能不成立）
- 字段映射有误导致 `profile.id` 未被赋值
- 渲染条件 `v-if="profile.id"` 不满足

建议检查：
1. 前端 `request.js` 响应拦截器的返回格式
2. `SettingsPage.vue` 中 `fetchProfile` 函数的数据解析逻辑
3. 响应数据结构与前端期望是否一致

### BUG-006：工具返回格式非JSON，前端解析失败

| 字段 | 内容 |
|------|------|
| **缺陷编号** | BUG-006 |
| **优先级** | P1（尽快修复） |
| **所属模块** | Day11 - Agent工具调用 |
| **影响范围** | 工具调用结果的前端展示效果 |

#### 缺陷现象
工具调用返回的结果是中文纯文本格式，前端尝试JSON解析失败。

控制台错误：
```
[tool_end] 解析失败: SyntaxError: Unexpected token '【', "【用户列表】（共 1... is not valid JSON
```

工具返回内容示例：
```
【用户列表】（共 106 人）
1. testuser（test@example.com）| 角色：无角色 | 超级管理员：否 | 最后登录：2026-07-17 15:12:46
2. review_user_20260707（review20260707@example.com）| 角色：无角色 | ...
...
```

#### 复现步骤
1. 进入智能对话页面
2. 发送"列出系统用户"等触发工具调用的消息
3. 打开浏览器控制台，查看tool_end事件

#### 当前进展
前端已增加降级处理：当JSON解析失败时，以纯文本形式展示工具结果。
用户体验影响：工具结果无法以结构化方式（表格、卡片等）展示。

#### 错误原因推断
Agent 工具节点的输出格式不是标准 JSON。工具本身可能返回的是自然语言文本而非结构化数据。

建议修复方向：
1. 规范所有工具的返回格式，统一使用 JSON 结构
2. 或在 Agent 的工具调用节点增加格式化步骤，将工具输出转换为 JSON
3. 工具返回格式示例：
   ```json
   {
     "tool": "list_users",
     "status": "success",
     "data": {
       "total": 106,
       "users": [...]
     },
     "summary": "共找到106个用户"
   }
   ```

### BUG-007：会话列表API返回401

| 字段 | 内容 |
|------|------|
| **缺陷编号** | BUG-007 |
| **优先级** | P2（择机修复） |
| **所属模块** | Day08 - 会话管理 |
| **影响范围** | 会话列表加载，用户体验 |

#### 缺陷现象
偶发性出现"获取会话列表失败：401"错误，刷新页面后恢复正常。

控制台错误：
```
获取会话列表失败: AxiosError: Request failed with status code 401
at fetchSessions (agent.js:36:16)
```

#### 错误原因推断
可能是 token 刷新机制或请求时序问题：
- 页面初始化时 `fetchSessions` 先于 token 写入 localStorage 发出
- 虽有401重试机制，但重试可能在某些情况下不生效
- 可能与 Pinia store 的初始化顺序有关

#### 影响评估
偶发性问题，刷新后可恢复，不影响核心功能使用，但影响用户体验。

### BUG-008：Element Plus Radio废弃警告

| 字段 | 内容 |
|------|------|
| **缺陷编号** | BUG-008 |
| **优先级** | P2（择机修复） |
| **所属模块** | 前端通用组件 |
| **影响范围** | 开发控制台警告，未来版本兼容性 |

#### 缺陷现象
控制台出现 Element Plus 废弃警告：
```
ElementPlusError: [el-radio] [API] label act as value is about to be deprecated in version 3.0.0, 
please use value instead.
```

#### 错误原因推断
项目中使用了 `el-radio` 的 `label` 属性作为单选框的值，但 Element Plus 新版本推荐使用 `value` 属性。

#### 影响评估
当前版本功能正常，仅为警告。升级到 Element Plus 3.0 时可能导致功能失效。

建议：将所有 `el-radio` / `el-radio-button` 的 `label` 属性改为 `value` 属性。

## 五、修复建议

### 针对 BUG-001 的修复思路

1. 检查 LangGraph 中工具节点的输出处理逻辑
2. 确保 tool call message 的 `function.arguments` 始终是有效的JSON字符串
3. 在调用 LLM 之前验证消息格式，增加防御性检查
4. 参考 OpenAI 的 tool calling 规范：`function.arguments` 必须是字符串化的JSON

关键检查点：
- `app/agents/` 目录下的工具定义和调用逻辑
- SSE 流中 `tool_call` 事件的数据格式
- 工具结果回传给 LLM 的消息构建逻辑
